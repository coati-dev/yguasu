  Question 1: Do you want this to be a standalone HTML file you open directly in

⏺ Good call — Vite on localhost is actually the right move here. Web MIDI API
  (the browser API that talks to your controller) requires a secure context, and
   localhost qualifies. A plain file:// HTML file would likely be blocked by the
   browser. So Vite it is.

  Question 2: Which browser do you primarily use? Web MIDI API has excellent
  support in Chrome/Edge but zero native support in Firefox (it requires an
  extension). This will affect what we need to build or warn about.

⏺ Perfect, Chrome it is — Web MIDI works natively there with no extra setup.

  Question 3: For the piano sounds, do you want sampled real piano audio (actual
   recordings of piano keys, very realistic but requires audio files) or
  synthesized sound (generated in the browser via Web Audio API, no files needed
   but more "electronic" sounding)? The synthesized route works out of the box
  with zero assets; the sampled route sounds much better but needs a soundfont
  or sample pack.

❯ how hard is it to change to a real piano audio after the initial prototype?   

⏺ Very easy — the audio playback would be isolated in one module. Swapping synth
   for samples later is essentially just replacing that one file/function. The
  rest of the app (MIDI detection, key display, UI) stays untouched.

  Question 4: Do you want a visual piano keyboard displayed on screen (so you
  can see which keys light up when you press them), or is a minimal UI with just

⏺ Not hard at all — MIDI gives you the exact note number, and mapping that to a
